# 业务事件详情可视化 PRD

## 1. 背景
当前“业务事件监控报表”列表页支持查看单条事件详情，但详情弹窗内展示的是原始 JSON 代码块。  
对于运营、售后、测试、研发等使用角色来说，虽然原始数据完整，但阅读成本高，存在以下问题：

- 用户需要自行理解 JSON 结构，无法快速看懂本次事件到底发生了什么
- 不同事件的 detail 结构差异较大，统一用代码块展示时缺少重点
- 遇到参数差异、流程失败等问题时，用户需要手动逐行排查，效率低
- 原始 JSON 适合作为排障兜底信息，但不适合作为主视图

本次需求目标是将事件 detail 从“原始 JSON 展示”升级为“结构化可视化展示”，在不丢失原始数据的前提下，提升事件详情的可读性与排查效率。

## 2. 目标
- 用户点击“查看明细”后，可直接看到结构化详情，而不是先读原始 JSON
- 针对不同 detail 结构，提供对应展示模板，减少用户自行解析成本
- 保留原始 JSON 作为低优先级兜底信息，满足排障查看需求
- 整体风格保持轻量、通用、可复用，不额外做重型分析页

## 3. 场景
### 场景 1：查看参数类事件详情
运营或测试人员在报表列表中查看某条自动校准事件，希望快速确认本地参数、计算参数、应用参数是否一致，而不是逐行对照 JSON 数组。

### 场景 2：查看简单结构事件详情
用户查看某些字段较少的事件，希望能直接看到几个关键字段，而不是弹出一个很空的代码块窗口。

### 场景 3：查看流程型事件详情
售后或研发查看一次数据拉取/刷新类事件，希望快速判断整次流程是否成功、失败发生在哪个分组、具体哪个步骤报错，并可在必要时查看原始上报 JSON。

## 4. 需求概述
在“业务事件监控报表”列表页中，点击行内“查看明细”后，打开事件详情弹窗。  
弹窗保留现有“大弹窗”形式，不改为抽屉。详情区域仅关注该事件自身内容，不重复列表页已承载的上下文摘要信息。

本次需求按 detail 数据结构提供三类结构化展示模板：

1. 参数对比型
2. 简单字段型
3. 分组流程型

同时在弹窗底部保留“原始 JSON”折叠区，默认收起，仅作为技术排障兜底层。

### 本次不做
- 不新增事件详情页独立路由，仍使用列表页内弹窗
- 不在详情弹窗顶部新增结果总结卡片、统计卡片、诊断结论卡片
- 不在本期做事件模板配置后台
- 不在本期支持用户自定义字段排序、字段隐藏、字段重命名
- 不对原始 JSON 做编辑、下载、二次处理，仅提供查看能力

## 5. 需求功能点

### 5.1 事件详情弹窗框架
**优先级：P0**

**页面 / 入口：**  
业务事件监控报表列表页，行内“查看明细”

**操作流程与页面改动：**
1. 用户点击“查看明细” → 页面打开事件详情弹窗
2. 弹窗标题统一为“事件详情”
3. 弹窗主体默认优先展示结构化详情内容
4. 弹窗底部保留“原始 JSON”折叠区，默认收起
5. 用户点击折叠区后 → 展开查看本次事件原始上报 JSON

**业务规则：**
- 弹窗样式保持当前大弹窗方案，不改为右侧抽屉
- 详情区聚焦事件 detail 内容，不重复展示列表页已有的大段上下文信息
- 原始 JSON 为兜底层，视觉优先级低于结构化详情
- 原始 JSON 文案统一为“查看原始 JSON”
- 原始 JSON 区域默认收起
- 原始 JSON 展开后应完整展示服务器返回的 detail 原文
- 原始 JSON 建议支持复制能力 `[待确认：是否本期一并支持复制按钮；若不支持，先只保留展开查看]`

**异常处理：**

| 异常场景 | 触发条件 | 系统行为 | 用户提示（草案，需 UX 确认） |
|----------|----------|----------|------|
| 明细为空 | detail 为空、null、undefined | 结构化区展示空态；原始 JSON 区仍保留 | 暂无事件明细 |
| 明细解析失败 | detail 不是合法 JSON，或结构化转换失败 | 不展示结构化区，直接回退到原始 JSON 视图 | 当前明细暂不支持结构化展示，请查看原始 JSON |
| 原始 JSON 为空 | 后端未返回原始内容 | 折叠区可保留但展开后显示空态 | 暂无原始数据 |

**边界条件：**
- detail 数据层级较深时，弹窗内容区需支持纵向滚动
- 原始 JSON 较长时，不影响上方结构化区阅读
- 展开原始 JSON 后，不改变已渲染的结构化区内容

### 5.2 参数对比型详情展示
**优先级：P0**

**适用范围：**  
适用于类似自动校准类事件，detail 中包含多组一一对应的参数数组，例如：
- `localParams`
- `serverParams`
- `finalWriteParams`

**页面 / 入口：**  
事件详情弹窗内，结构化详情区

**操作流程与页面改动：**
1. 系统识别到当前 detail 属于参数对比型结构 → 以表格方式展示
2. 页面按行展示每组参数位置对应的数据
3. 用户可直接横向对比“本地参数 / 计算参数 / 应用参数”

**业务规则：**
- 模块标题统一为“参数明细”
- 表格默认展示 4 列：
  - 参数序号
  - 本地参数
  - 计算参数
  - 应用参数
- 当数组存在时，以数组 index 顺序生成表格行
- 参数序号默认从业务侧给出的序列展示；若无业务序号映射，则按 1、2、3… 展示 `[待确认：现有 5/20/50/100/200/400 是否为固定业务序号，还是示意数据]`
- 数值按统一小数位格式展示 `[待确认：建议统一保留 3 位小数；若后端已格式化，则以前端收到值为准]`
- 若三组数组长度不一致，仍按最大长度生成行，缺失值显示 `—`
- 若只有部分字段存在（如只有 `localParams`、`serverParams`），则仅展示存在的列，不强依赖三列齐全

**异常处理：**

| 异常场景 | 触发条件 | 系统行为 | 用户提示（草案，需 UX 确认） |
|----------|----------|----------|------|
| 参数数组为空 | 数组为空或无有效元素 | 表格区显示空态 | 暂无参数数据 |
| 参数长度不一致 | 多组数组长度不同 | 缺失位置显示 `—`，不阻断展示 | 部分参数缺失 |
| 参数值非数字 | 数组内出现字符串、null、对象 | 原样展示或显示 `—` | 部分参数格式异常 |

**边界条件：**
- 表格列名需使用业务化文案，不直接暴露字段名
- 若后续新增同类事件，只要结构满足“多组同长度数组对比”，即可复用本模板
- 本期不强制提供差值列、高亮列 `[待确认：当前方案未包含差异高亮，如后续排查效率不足，再增量补充]`

### 5.3 简单字段型详情展示
**优先级：P1**

**适用范围：**  
适用于字段数量较少、结构简单的 detail，例如仅包含 `source` 等少量字段，或单层 key-value 数据。

**页面 / 入口：**  
事件详情弹窗内，结构化详情区

**操作流程与页面改动：**
1. 系统识别当前 detail 不属于参数对比型、流程型，但字段较少 → 使用简单字段型模板
2. 页面以紧凑描述列表方式展示字段名和值
3. 用户无需展开原始 JSON 即可看到关键信息

**业务规则：**
- 模块标题统一为“参数明细”
- 展示方式采用轻量 key-value 结构，不使用大块代码区
- 字段名展示使用业务文案映射，不直接暴露技术字段名 `[待确认：source 是否固定展示为“来源页面”]`
- 值为空时统一显示 `—`
- 单字段场景下，模块保持紧凑，不为少量字段制造大面积留白
- 若字段值为对象或数组，则不在本模板内继续展开复杂层级，应回退到通用 JSON 或对应专用模板

**异常处理：**

| 异常场景 | 触发条件 | 系统行为 | 用户提示（草案，需 UX 确认） |
|----------|----------|----------|------|
| 字段值为空 | 值为空字符串、null、undefined | 显示 `—` | - |
| 字段名无映射 | 未配置字段展示名 | 降级显示原字段名 | - |
| 字段结构超出简单模板范围 | 值为复杂对象/复杂数组 | 不按简单模板展示，回退其他模板 | 当前明细暂不支持该结构化视图 |

**边界条件：**
- 本模板目标是避免“字段很少但仍展示大段 JSON”的低效体验
- 本模板不做字段编辑、复制单字段值等增强能力

### 5.4 分组流程型详情展示
**优先级：P0**

**适用范围：**  
适用于 detail 中包含 `details[].groupName + tasks[]` 的流程型事件，例如数据获取、配置拉取、刷新类事件。

**页面 / 入口：**  
事件详情弹窗内，结构化详情区

**操作流程与页面改动：**
1. 系统识别当前 detail 包含分组流程结构 → 按 groupName 拆分为多个分组模块
2. 每个分组内使用表格展示步骤明细
3. 用户可按组查看每个步骤的执行结果、接口地址、描述信息

**业务规则：**
- 每个 `groupName` 渲染为一个独立区块
- 每个区块内表格默认展示以下列：
  - 步骤名称
  - 接口地址
  - 状态
  - 描述
- 步骤名称取 `name`
- 接口地址取 `path`
- 状态优先根据 `success` 判断：
  - `true` → 展示“成功”
  - `false` → 展示失败态
- 若存在明确错误码或失败 code，则状态列展示错误码；例如 `502`
- 描述列优先展示 `desc`；若 `desc` 为空但有错误信息，则展示错误信息
- 对于 `code`、`step` 等技术字段，本期不作为主列展示，避免增加阅读负担；如后续有排障需要，再考虑放入展开层 `[待确认]`
- 分组按后端返回顺序展示
- 失败步骤所在分组应允许用户直接看到失败项，不需要先读原始 JSON

**异常处理：**

| 异常场景 | 触发条件 | 系统行为 | 用户提示（草案，需 UX 确认） |
|----------|----------|----------|------|
| 分组为空 | `details` 为空数组 | 展示空态 | 暂无流程明细 |
| 某步骤字段缺失 | `name`、`path`、`desc` 等缺失 | 缺失字段显示 `—` | - |
| 接口地址过长 | `path` 超过单行可视范围 | 前端截断展示，支持完整查看 `[待确认：hover 还是点击展开]` | - |
| 存在失败步骤 | `success=false` 或返回错误码 | 失败状态显著展示 | - |
| overallSuccess 与步骤状态不一致 | 顶层状态与明细冲突 | 以步骤实际结果展示为准，不额外生成总结态文案 `[待确认：是否需要记录此类数据异常埋点]` | - |

**边界条件：**
- 分组数量较多时，整体区域支持纵向滚动
- 单组任务较多时，不分页，保持同一弹窗内连续阅读
- 状态展示以轻量标签为主，不扩展成流程图、时间轴等重型表达
- 本期不增加“重试”“跳转接口详情”“复制请求地址”等操作能力

### 5.5 模板识别与回退机制
**优先级：P0**

**页面 / 入口：**  
事件详情弹窗内，结构化详情区渲染前

**操作流程与页面改动：**
1. 系统获取 detail 数据
2. 按预设识别逻辑判断使用哪种模板
3. 若命中参数对比型，则渲染参数对比模板
4. 若命中分组流程型，则渲染分组流程模板
5. 若命中简单字段型，则渲染简单字段模板
6. 若均未命中，则回退到原始 JSON 展示方案

**业务规则：**
- 模板识别优先级建议为：
  1. 分组流程型
  2. 参数对比型
  3. 简单字段型
  4. 原始 JSON 回退
- `[待确认]` 若后端已有明确事件类型标识（如 eventCode / detailType），优先按事件类型匹配模板；若无，则按 detail 结构匹配
- 结构匹配只作为展示层判断，不修改原始数据
- 模板识别失败时，不能阻断详情弹窗打开

**异常处理：**

| 异常场景 | 触发条件 | 系统行为 | 用户提示（草案，需 UX 确认） |
|----------|----------|----------|------|
| 未命中任何模板 | detail 结构不符合已支持类型 | 回退原始 JSON | 当前明细暂不支持结构化展示 |
| 模板渲染异常 | 前端渲染时报错 | 回退原始 JSON，保证可查看 | 当前明细展示异常，请查看原始 JSON |

**边界条件：**
- 本期先覆盖已知高频结构，不要求一次性覆盖全部事件类型
- 后续新增模板时，不应影响原有模板判断逻辑
- 模板识别规则应可扩展，不写死在单一事件名称上 `[待确认：若业务后续稳定，建议逐步切换为“事件类型 + 结构”双重判断]`

## 6. 验收标准
- 当用户在业务事件监控报表列表中点击“查看明细”时，可打开事件详情弹窗
- 当事件 detail 为参数对比型数据时，页面以表格形式展示参数明细，而不是原始 JSON 代码块
- 当事件 detail 为简单字段型数据时，页面以紧凑字段列表展示，而不是出现大面积空白或仅展示代码块
- 当事件 detail 为分组流程型数据时，页面按分组展示步骤明细，用户可直接查看每个步骤的状态、接口地址和描述
- 当事件 detail 结构未命中任何已支持模板时，页面仍可通过原始 JSON 折叠区查看内容，不影响用户排障
- 原始 JSON 在详情弹窗底部默认收起，用户点击后可展开查看完整内容
- 当 detail 为空、字段缺失、结构异常时，页面不报错，能够给出空态或回退展示

## 待确认项
1. 参数对比型中的“参数序号”是否有固定业务含义，是否需要展示为 5/20/50/100/200/400，而不是按数组 index 展示。
2. 参数值展示是否统一保留 3 位小数，还是直接沿用后端返回格式。
3. 简单字段型中 `source` 是否固定映射为“来源页面”，以及后续是否有其他固定字段映射表。
4. 分组流程型中，接口地址的完整查看方式采用 hover、点击展开，还是换行展示。
5. 分组流程型中，是否需要保留 `step` / `code` 等技术字段的次级查看入口。
6. 若后端已有明确事件类型标识，详情模板是否按事件类型优先匹配，而不是仅按 JSON 结构匹配。
7. 原始 JSON 区域是否本期同步支持“复制 JSON”按钮。
